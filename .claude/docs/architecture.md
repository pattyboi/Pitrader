# Architecture

Referenced from `CLAUDE.md`. Eight supporting modules plus the one strategy class (`AssetRotationStrategy` in `strategy.py`), all in `pi-trading-agent/`.

- `main.py` — loads and strictly validates every key in `config.json` (`load_config` raises on any missing/invalid value). Core keys (Alpaca credentials, asset symbols, dip threshold, email, news, LLM) sit in a strict `required` set that raises if absent. Later feature groups (portfolio, decision memory, symbol reference, autonomous discovery) instead get a `setdefault`-style defaults dict so older configs keep working after a feature ships, then the same range/type validation runs on the defaulted value. A new key belongs in `required` only if there's no sane default; otherwise add it to the matching defaults dict, validate it, and thread it into the strategy `parameters` dict. Builds the Lumibot `Alpaca` broker and `Trader`, and runs `AssetRotationStrategy`.
- `strategy.py` — `AssetRotationStrategy(Strategy)` with `sleeptime = "1D"`. Portfolio mode is the only mode: `on_trading_iteration` is a thin wrapper around `_run_portfolio_iteration`, the whole decision pipeline (see `decision-pipeline.md`). `on_filled_order` completes in-flight rotations/exits.
- `news_context.py` — `WorldEventAnalyzer` fetches Alpaca news and produces a deterministic keyword score (`SEVERE_RISK_TERMS` −3, `RISK_TERMS` −1, `OPPORTUNITY_TERMS` +1, each counted once per article) wrapped in a `NewsContext`, which also carries the raw articles and a per-symbol score breakdown (from Alpaca's own article `symbols` tags) for downstream analysis. The pure `score_articles` classmethod (article dicts in, no network) does the scoring and is what `analyze()` calls after fetching; `NEWS_SCORE_REFINEMENT_ENABLED` (off by default) turns on recency decay and duplicate-phrase dampening inside it — off reproduces the original unweighted score exactly, so this can't silently change `NEWS_HIGH_RISK_SCORE`'s behavior.
- `symbol_reference.py` — `SymbolReference`, a local ticker→company-name mapping cross-checked from Alpaca's per-symbol asset lookup and the SEC's public `company_tickers.json`, persisted in `.symbol_reference.duckdb` and refreshed at most every `SYMBOL_REFERENCE_REFRESH_DAYS`. `verified_symbols()` filters a spurious/malformed Alpaca symbol tag before it's trusted (empty result — nothing cached yet — means fail open, not reject-everything); `scan_text_for_symbols()` is a bounded company-name text scan, scoped only to the day's candidates, that catches a mention Alpaca's own tagging missed. Never creates a trade or veto on its own.
- `llm_news.py` — optional `LLMNewsAnalyzer` sends the same articles to a local LLM (one call per trading day, score −10…+10). Local-only by design: the only supported backend is a local Ollama server's OpenAI-compatible endpoint (`LLM_NEWS_BASE_URL`, default `http://127.0.0.1:11434/v1`) — no other provider is supported, so this assessment never depends on an outside service or leaves the box. Default model is `phi3:mini`, whose ~4k-token context window (input + output combined) is far smaller than a hosted model's, so every prompt-assembling call (`assess`, `explain_exit`, `check_red_flag`) runs its articles through `_prioritize_articles` first: a ~4-chars/token estimate (`_estimate_tokens`, no tokenizer dependency for Pi-constrained installs) selects the highest-`|score|` articles that fit `MAX_PROMPT_TOKENS_ESTIMATE` (1500), then restores original order before formatting — never truncates an individual article's text (`MAX_SUMMARY_CHARS`'s job), only trims which whole articles are included; `assess()` additionally reserves budget for its symbol-context block before spending the rest on articles, and fails open to the unfiltered list if prioritization itself raises. `MAX_RESPONSE_TOKENS` (500) is capped accordingly, leaving headroom in the small context window for the ~1500-token input. Gets the JSON contract via prompt + `response_format: json_object`, and replies are repaired defensively (`_parse_assessment`: fence-stripping, score clamping, risk-level derivation). Disabled by default. Advisory unless `LLM_NEWS_BLOCK_ON_HIGH_RISK` is true. `ollama.service` (installed by `setup_service.sh`, binds loopback-only, `OLLAMA_KEEP_ALIVE=1h`) and `ollama-warmup.timer` (fires 09:00 ET via `scripts/ollama_warmup.sh`, 30 minutes ahead of the 9:30 ET market open) keep the model loaded for the single daily call this makes (`sleeptime = "1D"`, so there is exactly one `on_trading_iteration` per day) without paying a cold-start cost during the trading iteration; the 1h window covers the 9:00 warm-up through the 9:30 call with buffer, then the model unloads on its own instead of holding ~3GB of RAM the other 23 hours. `assess()` also takes today's evaluation universe (`symbols`, `held_symbols`, `symbol_scores` from `_symbol_news_scores` — the same cross-checked per-symbol map ranking uses, computed once in `_run_portfolio_iteration` and reused for both) so the model reasons about risk to the specific symbols this agent might trade today, not just generic market headlines; the score it returns still measures aggregate market risk only, since that's all `_market_veto_reason` consumes. Three more methods reuse the same warm model for purely descriptive or advisory work, never a new decision: `summarize_day()` (a 2-3 sentence recap of the report, `strategy.py`'s `_generate_daily_narrative`, shown at the top of the daily email), `explain_exit()` (one sentence connecting a just-submitted take-profit/stop-loss/horizon exit to that symbol's own coverage, `_generate_exit_narrative`, logged as an extra `portfolio_actions` line — the sale already fired on price alone before this runs), and `check_red_flag()` (screens a discovery-only candidate — never a held/static symbol — with negative dedicated coverage for a severe company-specific risk the liquidity floor can't see; `_check_discovery_red_flags` only spends a call when `symbol_news_scores` is already negative for that symbol, logs+reports a flag either way, and only actually excludes the symbol from `symbols` when `PORTFOLIO_DISCOVERY_LLM_BLOCK_ENABLED` is true — advisory-by-default like `LLM_NEWS_BLOCK_ON_HIGH_RISK`). All four share `LLMNewsAnalyzer._chat()`; the three narrative-style calls return plain text (`_clean_narrative`: fence/quote stripping, length cap) rather than JSON.
- `article_filter.py` — standalone `extract_financial_context(url, watchlist)`, wired into `strategy.py`'s `_check_discovery_article_context` right alongside `_check_discovery_red_flags` (same trigger: a discovery-only candidate with negative dedicated coverage; same fail-open, advisory-only, logged+reported posture — never excludes a symbol on its own). Unlike the rest of the news stack, this fetches the candidate's own linked article in full (`trafilatura.fetch_url`/`extract`, not just Alpaca's headline/summary — `news_context.py`'s `per_article` now carries a `url` field for this), sentence-scores it against the watchlist and a fixed financial-signal-term set, keeps only the top 15 sentences (restored to original order), and skips articles too thin to be worth a call (`MIN_TOKENS`=80) or truncates ones too long (`MAX_TOKENS`=800, at a sentence boundary) before asking the same local model (via Ollama's `/api/generate`, not the `/v1/chat/completions` endpoint `llm_news.py` uses) for a structured `{sentiment, confidence, affected_tickers, key_risks, catalyst_type}` verdict. Results are cached per `(url, calendar day)` in `.article_cache.json` so a re-surfaced candidate doesn't refetch. Every failure (fetch, extraction, HTTP, JSON parsing) is caught inside `extract_financial_context` itself and turned into `None` — callers never need their own try/except.
- `adaptive_news_model.py` — `AdaptiveNewsModel`, a ridge-stabilized one-variable regression of next-day Asset B return on the news score, persisted in `.news_learning_state.json`. It only becomes authoritative after `NEWS_LEARNING_MIN_OBSERVATIONS` samples *and* sufficient `|correlation|`. Learns from Asset B as a market proxy inside portfolio mode.
- `trade_memory.py` — `TradeMemory`, a DuckDB journal (`.trade_memory.duckdb`, migrated once from a legacy `.trade_memory.sqlite3` if present) plus a two-feature ridge regression of the next-session B-minus-A edge on dip size and news score. This is the engine behind the "Opportunistic Opportunity" — portfolio mode's occasional A-to-B swap (`_opportunistic_opportunity`), trained only on settled A-versus-B observations; it is not a separate trading mode. Unsettled observations older than `MAX_SETTLEMENT_GAP_DAYS` (4 calendar days) are never settled — a longer gap would record a multi-day return as one session. Every call site opens/closes its own `duckdb.connect()`; concurrent access from the broker-callback thread (`on_filled_order`) and the main iteration thread is possible and can raise `TransactionException: Catalog write-write conflict` — every call site must wrap in its own `try/except`, not just rely on the outer iteration handler.
- `autonomous_universe.py` — `AutonomousUniverse`, bounded daily discovery over Alpaca's asset directory, persisted in `.autonomous_universe.duckdb` (migrated once, on first use after upgrade, from a legacy `.autonomous_universe.json` if present — mirrors `trade_memory.py`'s sqlite-to-duckdb migration pattern; the old file is never deleted). The assets host follows the trading mode (paper vs live keys are host-specific). `remember()` deduplicates keeping the most recent mention (via a monotonically increasing `last_seen_rank`) so re-confirmed symbols (e.g. current holdings) are never trimmed as stale. Discovered symbols are still subject to `_portfolio_signal`'s price/volume liquidity floor (see `decision-pipeline.md`) before they can ever be bought.
- `portfolio_memory.py` — `PortfolioMemory`, a DuckDB journal (`.portfolio_memory.duckdb`) keyed `(evaluation_date, symbol)`, unlike `trade_memory.py`'s single A/B pair. Every symbol the portfolio evaluates each day — not just one with a qualifying dip signal today — contributes a settled observation carrying at least five learned facts (dip %, per-symbol news score, live bid/ask spread, recent average volume, and the historical backtest edge/win-probability/return-stdev), keyed by a `signal_present` column recording whether today's dip actually cleared the live threshold. One **pooled** two-feature ridge regression (dip size + news score → next-session return) is fit only across `signal_present = 1` history, exactly like `trade_memory.py`'s own `signal_present` column — broadening daily coverage to every watched/held symbol grows the durable context without diluting the forecast with ordinary non-dip market days. Feeds `_posture_adjusted_edge`'s ranking only (see `decision-pipeline.md`), never eligibility. Same settlement-window and price-only-backfill conventions as `trade_memory.py`, but settlement is scoped to each symbol's own later price.
