# Architecture

Referenced from `CLAUDE.md`. Eight supporting modules plus the one strategy class (`AssetRotationStrategy` in `strategy.py`), all in `pi-trading-agent/`.

- `main.py` ‚Äî loads and strictly validates every key in `config.json` (`load_config` raises on any missing/invalid value). Core keys (Alpaca credentials, asset symbols, dip threshold, email, news, LLM) sit in a strict `required` set that raises if absent. Later feature groups (portfolio, decision memory, symbol reference, autonomous discovery) instead get a `setdefault`-style defaults dict so older configs keep working after a feature ships, then the same range/type validation runs on the defaulted value. A new key belongs in `required` only if there's no sane default; otherwise add it to the matching defaults dict, validate it, and thread it into the strategy `parameters` dict. `build_strategy(config, base_dir)` builds the Lumibot `Alpaca` broker and the `AssetRotationStrategy` parameters dict without starting the `Trader` loop, so `main()` can wrap it in `Trader().add_strategy(...).run_all()` while `scripts/nightly_preeval.py` (see `decision-pipeline.md`'s nightly pre-evaluation side channel) reuses the exact same construction and calls one strategy method directly instead ‚Äî kept as one function so the two entrypoints' parameters dicts can never drift apart as config keys are added.
- `strategy.py` ‚Äî `AssetRotationStrategy(Strategy)` with `sleeptime = "10M"` (a cheap poll, not the evaluation cadence ‚Äî see `decision-pipeline.md`'s cadence note for `_due_iteration_window_now`/`_due_portfolio_iteration_window`). Portfolio mode is the only mode: `on_trading_iteration` gates on that due-window check and, at most twice a trading day, calls `_run_portfolio_iteration`, the whole decision pipeline (see `decision-pipeline.md`). `on_filled_order` completes in-flight rotations/exits. `on_abrupt_closing` dumps every live thread's stack (`faulthandler.dump_traceback`) to `.shutdown_diagnostic.log` the instant a stop signal arrives ‚Äî Lumibot calls it synchronously in the main thread right when `StrategyExecutor.stop()` sets its `stop_event`, before any join or systemd `TimeoutStopSec` countdown, so a mid-day `systemctl stop` that still needs a SIGKILL despite `main.py`'s interruptible market-open/market-close waits leaves real evidence of what was actually blocking instead of only a restart in the journal.
- `news_context.py` ‚Äî `WorldEventAnalyzer` fetches Alpaca news and produces a deterministic keyword score (`SEVERE_RISK_TERMS` ‚àí3, `RISK_TERMS` ‚àí1, `OPPORTUNITY_TERMS` +1, each counted once per article) wrapped in a `NewsContext`, which also carries the raw articles and a per-symbol score breakdown (from Alpaca's own article `symbols` tags) for downstream analysis. The pure `score_articles` classmethod (article dicts in, no network) does the scoring and is what `analyze()` calls after fetching; `NEWS_SCORE_REFINEMENT_ENABLED` (off by default) turns on recency decay and duplicate-phrase dampening inside it ‚Äî off reproduces the original unweighted score exactly, so this can't silently change `NEWS_HIGH_RISK_SCORE`'s behavior.
- `symbol_reference.py` ‚Äî `SymbolReference`, a local ticker‚Üícompany-name mapping cross-checked from Alpaca's per-symbol asset lookup and the SEC's public `company_tickers.json`, persisted in `.symbol_reference.duckdb` and refreshed at most every `SYMBOL_REFERENCE_REFRESH_DAYS`. `verified_symbols()` filters a spurious/malformed Alpaca symbol tag before it's trusted (empty result ‚Äî nothing cached yet ‚Äî means fail open, not reject-everything); `scan_text_for_symbols()` is a bounded company-name text scan, scoped only to the day's candidates, that catches a mention Alpaca's own tagging missed. Never creates a trade or veto on its own.
- `llm_news.py` ‚Äî optional `LLMNewsAnalyzer` sends the same articles to a local LLM (up to two calls per trading day, score ‚àí10‚Ä¶+10). Local-only by design: the only supported backend is a local Ollama server's OpenAI-compatible endpoint (`LLM_NEWS_BASE_URL`, default `http://127.0.0.1:11434/v1`) ‚Äî no other provider is supported, so this assessment never depends on an outside service or leaves the box. Default model is `hf.co/unsloth/granite-4.0-micro-GGUF:Q4_K_M` (trains natively at 131072 tokens), but Ollama silently serves any API request at a ~4096-token context by default regardless of what the model supports (confirmed directly in this model's own server log: `"vram-based default context" default_num_ctx=4096`) unless a request explicitly asks for more, so `_chat()` sets `options.num_ctx` to `OLLAMA_NUM_CTX` (8192). That reclaimed headroom is not, by itself, license to fill the prompt anywhere near that much: this Pi's CPU-only inference runs ~27-31 tokens/sec on prompt-eval and ~4 tokens/sec on generation *regardless of which ~3-4B model is configured* (measured directly against both `llama3.2:3b` and the current model), so every prompt-assembling call (`assess`, `explain_exit`, `check_red_flag`) still runs its articles through `_prioritize_articles` first: a ~4-chars/token estimate (`_estimate_tokens`, no tokenizer dependency for Pi-constrained installs ‚Äî deliberately sized against the *worse* of two measured cases, since it undercounted one model's real tokens by ~1.6x and overcounted another's by ~1.1x for the same content) selects the highest-`|score|` articles that fit `MAX_PROMPT_TOKENS_ESTIMATE` (1800), then restores original order before formatting ‚Äî never truncates an individual article's text (`MAX_SUMMARY_CHARS`'s job), only trims which whole articles are included. All three calls reserve budget for their own fixed system prompt (and `assess()` additionally for its symbol-context block) before spending the rest on articles ‚Äî a full day's headlines were once observed overflowing the intended budget in production because this reservation was missing entirely ‚Äî and `assess()` fails open to the unfiltered list if prioritization itself raises. `MAX_RESPONSE_TOKENS` (400) and `REQUEST_TIMEOUT_SECONDS` (240, generous since this is at most a twice-a-day background call, not a latency-sensitive one) are both sized against those measured tokens/sec figures so a worst-case prompt still finishes with margin. Gets the JSON contract via prompt + `response_format: json_object`, and replies are repaired defensively (`_parse_assessment`: fence-stripping, score clamping, risk-level derived from score rather than trusted from the model's own label ‚Äî a self-reported level that disagreed with its own score once shipped silently, see git history). Disabled by default. Advisory unless `LLM_NEWS_BLOCK_ON_HIGH_RISK` is true. `ollama.service` (installed by `setup_service.sh`, binds loopback-only, `OLLAMA_KEEP_ALIVE=8h`, `OLLAMA_MAX_LOADED_MODELS=1` ‚Äî a hard cap on this memory-constrained Pi, not a reaction to any current multi-model use, since every caller here already shares the one configured tag) and `ollama-warmup.timer` (fires 09:00 ET via `scripts/ollama_warmup.sh`, 30 minutes ahead of the 9:30 ET market open) keep the model loaded for both of the day's calls (`sleeptime = "10M"`, gated to at most two `on_trading_iteration` runs a day ‚Äî see `decision-pipeline.md`) without paying a cold-start cost during either one; the 8h window covers the 9:00 warm-up through both the `"open"` call and the `"midday"` call (`PORTFOLIO_SECOND_ITERATION_OFFSET_MINUTES` after open, default 210) regardless of how that offset is configured, then the model unloads on its own overnight instead of holding ~2GB of RAM around the clock. `assess()` also takes today's evaluation universe (`symbols`, `held_symbols`, `symbol_scores` from `_symbol_news_scores` ‚Äî the same cross-checked per-symbol map ranking uses, computed once in `_run_portfolio_iteration` and reused for both) so the model reasons about risk to the specific symbols this agent might trade today, not just generic market headlines; the score it returns still measures aggregate market risk only, since that's all `_market_veto_reason` consumes. Three more methods reuse the same warm model for purely descriptive or advisory work, never a new decision: `summarize_day()` (a 2-3 sentence recap of the report, `strategy.py`'s `_generate_daily_narrative`, shown at the top of the daily email), `explain_exit()` (one sentence connecting a just-submitted take-profit/stop-loss/horizon exit to that symbol's own coverage, `_generate_exit_narrative`, logged as an extra `portfolio_actions` line ‚Äî the sale already fired on price alone before this runs), and `check_red_flag()` (screens a discovery-only candidate ‚Äî never a held/static symbol ‚Äî with negative dedicated coverage for a severe company-specific risk the liquidity floor can't see; `_check_discovery_red_flags` only spends a call when `symbol_news_scores` is already negative for that symbol, logs+reports a flag either way, and only actually excludes the symbol from `symbols` when `PORTFOLIO_DISCOVERY_LLM_BLOCK_ENABLED` is true ‚Äî advisory-by-default like `LLM_NEWS_BLOCK_ON_HIGH_RISK`). All four share `LLMNewsAnalyzer._chat()`; the three narrative-style calls return plain text (`_clean_narrative`: fence/quote stripping, length cap) rather than JSON.
- `article_filter.py` ‚Äî standalone `extract_financial_context(url, watchlist)`, wired into `strategy.py`'s `_check_discovery_article_context` right alongside `_check_discovery_red_flags` (same trigger by default: a discovery-only candidate with negative dedicated coverage; same fail-open, advisory-only, logged+reported posture ‚Äî never excludes a symbol on its own). Unlike the rest of the news stack, this fetches the candidate's own linked article in full (`trafilatura.fetch_url`/`extract`, not just Alpaca's headline/summary ‚Äî `news_context.py`'s `per_article` now carries a `url` field for this), sentence-scores it against the watchlist and a fixed financial-signal-term set, keeps only the top 15 sentences (restored to original order), and skips articles too thin to be worth a call (`MIN_TOKENS`=80) or truncates ones too long (`MAX_TOKENS`=800, at a sentence boundary) before asking the same local model (via Ollama's `/api/generate`, not the `/v1/chat/completions` endpoint `llm_news.py` uses) for a structured `{sentiment, confidence, affected_tickers, key_risks, catalyst_type}` verdict. Results are cached per `(calendar day, url, digest-of-watchlist)` in `.article_verdicts.duckdb` (migrated from a legacy `.article_cache.json`) so a re-surfaced candidate doesn't refetch. Every failure (fetch, extraction, HTTP, JSON parsing) is caught inside `extract_financial_context` itself and turned into `None` ‚Äî callers never need their own try/except. `_check_discovery_article_context` takes a `require_negative_score` flag (default `True`, the live discovery-only behavior above); `scripts/nightly_preeval.py`'s `_run_nightly_preevaluation` (see `decision-pipeline.md`) calls it with `False` over every managed/held symbol once nightly at 03:00 ET, so this same cache is pre-warmed before the live iteration ever asks ‚Äî a cache hit there skips the Ollama round-trip entirely.
- `adaptive_news_model.py` ‚Äî `AdaptiveNewsModel`, a ridge-stabilized one-variable regression of next-day Asset B return on the news score. It only becomes authoritative after `NEWS_LEARNING_MIN_OBSERVATIONS` samples *and* sufficient `|correlation|`. Learns from Asset B as a market proxy inside portfolio mode. `strategy.py` instantiates it twice with separate state files: `_update_adaptive_learning` trains it on the deterministic keyword `news_context.score` (`.news_learning_state.json`, this is the one `_market_veto_reason` actually consumes via `NEWS_PREDICTED_RETURN_BLOCK_PERCENT`); `_update_llm_adaptive_learning` trains a second instance on `llm_assessment.score` (`.news_learning_state_llm.json`), purely so the two signals' predictive value can be compared over settled outcomes ‚Äî its `llm_learned_forecast` is reported but not (yet) wired into any veto. Each instance keys one observation per calendar date regardless of how many times `on_trading_iteration` runs that day (`AdaptiveNewsModel.update`'s own same-day `pending` guard), so the twice-daily cadence does not double-weight a day's observation.
- `trade_memory.py` ‚Äî `TradeMemory`, a DuckDB journal (`.trade_memory.duckdb`, migrated once from a legacy `.trade_memory.sqlite3` if present) plus a two-feature ridge regression of the next-session B-minus-A edge on dip size and news score. This is the engine behind the "Opportunistic Opportunity" ‚Äî portfolio mode's occasional A-to-B swap (`_opportunistic_opportunity`), trained only on settled A-versus-B observations; it is not a separate trading mode. Unsettled observations older than `MAX_SETTLEMENT_GAP_DAYS` (4 calendar days) are never settled ‚Äî a longer gap would record a multi-day return as one session. Every call site opens/closes its own `duckdb.connect()`; concurrent access from the broker-callback thread (`on_filled_order`) and the main iteration thread is possible and can raise `TransactionException: Catalog write-write conflict` ‚Äî every call site must wrap in its own `try/except`, not just rely on the outer iteration handler.
- `autonomous_universe.py` ‚Äî `AutonomousUniverse`, bounded daily discovery over Alpaca's asset directory, persisted in `.autonomous_universe.duckdb` (migrated once, on first use after upgrade, from a legacy `.autonomous_universe.json` if present ‚Äî mirrors `trade_memory.py`'s sqlite-to-duckdb migration pattern; the old file is never deleted). The assets host follows the trading mode (paper vs live keys are host-specific). `remember()` deduplicates keeping the most recent mention (via a monotonically increasing `last_seen_rank`) so re-confirmed symbols (e.g. current holdings) are never trimmed as stale. Discovered symbols are still subject to `_portfolio_signal`'s price/volume liquidity floor (see `decision-pipeline.md`) before they can ever be bought. A symbol that passes Alpaca's `tradable`/`fractionable` asset filter but comes back with zero historical bars (e.g. a thinly traded OTC ADR) can never clear that floor either, so `_portfolio_signal` reports it and `_run_portfolio_iteration` calls `exclude_unpriceable()` for any such symbol that came from discovery (never a config-listed watchlist symbol, which stays eligible through a transient data outage) ‚Äî persisted in `unpriceable_symbols` and filtered out of every future `next_batch()` result so the same dead symbol doesn't resurface and re-fail once the rotation cursor wraps back to it.
- `portfolio_memory.py` ‚Äî `PortfolioMemory`, a DuckDB journal (`.portfolio_memory.duckdb`) keyed `(evaluation_date, symbol)`, unlike `trade_memory.py`'s single A/B pair. Every symbol the portfolio evaluates each day ‚Äî not just one with a qualifying dip signal today ‚Äî contributes a settled observation carrying at least five learned facts (dip %, per-symbol news score, live bid/ask spread, recent average volume, and the historical backtest edge/win-probability/return-stdev), keyed by a `signal_present` column recording whether today's dip actually cleared the live threshold. One **pooled** two-feature ridge regression (dip size + news score ‚Üí next-session return) is fit only across `signal_present = 1` history, exactly like `trade_memory.py`'s own `signal_present` column ‚Äî broadening daily coverage to every watched/held symbol grows the durable context without diluting the forecast with ordinary non-dip market days. Feeds `_posture_adjusted_edge`'s ranking only (see `decision-pipeline.md`), never eligibility. Same settlement-window and price-only-backfill conventions as `trade_memory.py`, but settlement is scoped to each symbol's own later price.
