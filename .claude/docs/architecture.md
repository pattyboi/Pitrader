# Architecture

Referenced by `/mnt/dietpi_userdata/staging/pi-trading/CLAUDE.md`. Keep this in sync with any module/behavior change.

## Two independent processes

The repo runs as **two separate systemd services**, each its own OS process:

- `main.py` → `strategy.py`'s `AssetRotationStrategy` — equities, gated to run around NYSE open (twice a trading day).
- `main_crypto.py` → `crypto_strategy.py`'s `CryptoRotationStrategy` — crypto, gated to run only while NYSE is *closed*.

They are separate processes, not one process with two Lumibot strategies, because Lumibot's `Trader.add_strategy()` raises `NotImplementedError` for a second live strategy in one `Trader` (`lumibot/traders/trader.py`). Two processes also mean a crash or restart in one can never take the other down. Both read the same `config.json` and share the same Alpaca account/credentials, but write only their own, fully disjoint state files (`.portfolio_*`/`.trade_memory.duckdb`/etc. for equity, `.crypto_*` for crypto).

## Module map

**Entry points**
- `main.py` — loads and strictly validates every key in `config.json` (`load_config` raises on any missing/invalid value). Core keys (Alpaca credentials, asset symbols, dip threshold, email, news, LLM) sit in a strict `required` set that raises if absent. Later feature groups (portfolio, decision memory, symbol reference, autonomous discovery, crypto) instead get a `setdefault`-style defaults dict so older configs keep working after a feature ships, then the same range/type validation runs on the defaulted value. A new key belongs in `required` only if there's no sane default; otherwise add it to the matching defaults dict, validate it, and thread it into the strategy `parameters` dict. `build_strategy(config, base_dir)` builds the Lumibot `Alpaca` broker and the `AssetRotationStrategy` parameters dict without starting the `Trader` loop, so `main()` can wrap it in `Trader().add_strategy(...).run_all()` while `scripts/nightly_preeval.py` (see `decision-pipeline.md`'s nightly pre-evaluation side channel) reuses the exact same construction and calls one strategy method directly instead — kept as one function so the two entrypoints' parameters dicts can never drift apart as config keys are added. Also defines `MarketOpenLoggingAlpaca`, an `Alpaca` broker subclass that caches `market_hours()` per calendar day (avoids a CPU-pegging bug from Lumibot recomputing the NYSE holiday calendar on every poll tick) and makes market-open/close waits interruptible so systemd stops cleanly.
- `main_crypto.py` — parallel entry point for crypto. Reuses `main.load_config`/`MarketOpenLoggingAlpaca` rather than duplicating validation; only `build_crypto_strategy` and its own `main()` are crypto-specific.

**Strategy classes**
- `strategy.py` — `AssetRotationStrategy(Strategy)`, ~3300 lines, with `sleeptime = "10M"` (a cheap poll, not the evaluation cadence — see `decision-pipeline.md`'s cadence note for `_due_iteration_window_now`/`_due_portfolio_iteration_window`). Portfolio mode is the only mode: `on_trading_iteration` gates on that due-window check and, at most twice a trading day, calls `_run_portfolio_iteration`, the whole decision pipeline (see `decision-pipeline.md`). `on_filled_order` completes in-flight rotations/exits. `on_abrupt_closing` dumps every live thread's stack (`faulthandler.dump_traceback`) to `.shutdown_diagnostic.log` the instant a stop signal arrives — Lumibot calls it synchronously in the main thread right when `StrategyExecutor.stop()` sets its `stop_event`, before any join or systemd `TimeoutStopSec` countdown, so a mid-day `systemctl stop` that still needs a SIGKILL despite `main.py`'s interruptible market-open/market-close waits leaves real evidence of what was actually blocking instead of only a restart in the journal.
- `crypto_strategy.py` — `CryptoRotationStrategy(Strategy)`. Deliberately its own class, not a subclass of `AssetRotationStrategy` — inheriting would couple crypto to equity-specific internals (email HTML, equity-tuned discovery heuristics) that would need silent overriding. Reimplements the same decision shape (dip signal, exits, build, discovery, Opportunistic Opportunity swap) narrowly for crypto, sharing only genuinely asset-class-agnostic code (below). No news/LLM integration — crypto has none of that layer yet.

**Shared, asset-class-agnostic modules** (imported by both strategy classes)
- `decision_math.py` — pure math with zero broker/`self` coupling: `walk_forward_net_returns` (chronological out-of-sample validation), `posture_adjusted_edge` (risky/conservative ranking reshaping), `optimal_position_count` (Sharpe-like position sizing). `AssetRotationStrategy` keeps its old method names (`_walk_forward_net_returns` etc.) as `staticmethod` aliases onto these functions so existing call sites/tests are unaffected.
- `email_render.py` — generic HTML-table-rendering helpers (`email_kv_section`, `email_bullet_section`, `email_status_theme`, `render_email_shell`) extracted from `strategy.py`'s email code. Both strategies build their own report-specific sections and pass them in; this module has no report-shape assumptions.
- `market_sessions.py` — `is_next_trading_session` (NYSE-session succession, used by equity's memory classes), `nyse_is_open` (cached NYSE-open predicate, used by crypto's gating — see `decision-pipeline.md`), `is_next_calendar_day` (plain calendar-day succession, used by crypto's memory classes since crypto trades every day, not just NYSE sessions).
- `trade_memory.py` — `TradeMemory`, a DuckDB journal (`.trade_memory.duckdb`) plus a two-feature ridge regression (the shared `ridge_regression.fit_two_feature_ridge`, also used by `portfolio_memory.py`) of the next-session B-minus-A edge on dip size and news score. This is the engine behind the "Opportunistic Opportunity" — portfolio mode's occasional A-to-B swap (`_opportunistic_opportunity`), trained only on settled A-versus-B observations; it is not a separate trading mode. Unsettled observations older than `MAX_SETTLEMENT_GAP_DAYS` (4 calendar days) are never settled — a longer gap would record a multi-day return as one session. Both strategies keep one `TradeMemory` instance cached per `(database_path, minimum_observations, maximum_observations)` key (`_decision_memory()`/`_crypto_decision_memory()`) rather than constructing a fresh one at every call site, so schema creation (`_schema_initialized`) only runs once per key; each method call still opens/closes its own `duckdb.connect()`, so concurrent access from the broker-callback thread (`on_filled_order`) and the main iteration thread is still possible and can still raise `TransactionException: Catalog write-write conflict` — every call site must still wrap in its own `try/except`, not just rely on the outer iteration handler. Accepts a `next_session_predicate` constructor parameter (default `is_next_trading_session`, preserving equity's exact behavior); `CryptoRotationStrategy` passes `is_next_calendar_day` instead — using the NYSE-session default for crypto would silently fail to settle a Friday observation until the following Monday, spanning a weekend of real crypto price action as if it were a single "next session" return.
- `portfolio_memory.py` — `PortfolioMemory`, a DuckDB journal (`.portfolio_memory.duckdb`) keyed `(evaluation_date, symbol)`, unlike `trade_memory.py`'s single A/B pair. Every symbol the portfolio evaluates each day — not just one with a qualifying dip signal today — contributes a settled observation carrying at least five learned facts (dip %, per-symbol news score, live bid/ask spread, recent average volume, and the historical backtest edge/win-probability/return-stdev), keyed by a `signal_present` column recording whether today's dip actually cleared the live threshold. One **pooled** two-feature ridge regression (`ridge_regression.fit_two_feature_ridge`, shared with `trade_memory.py`; dip size + news score → next-session return) is fit only across `signal_present = 1` history, exactly like `trade_memory.py`'s own `signal_present` column — broadening daily coverage to every watched/held symbol grows the durable context without diluting the forecast with ordinary non-dip market days. Feeds `_posture_adjusted_edge`'s ranking only (see `decision-pipeline.md`), never eligibility. Same settlement-window and price-only-backfill conventions as `trade_memory.py`, but settlement is scoped to each symbol's own later price. Takes the same `next_session_predicate` parameterization as `trade_memory.py` above.
- `autonomous_universe.py` — `AutonomousUniverse`, bounded daily discovery over Alpaca's asset directory, persisted in a per-instance DuckDB file (`.autonomous_universe.duckdb` for equity, `.crypto_universe.duckdb` for crypto). The assets host follows the trading mode (paper vs live keys are host-specific). `remember()` deduplicates keeping the most recent mention (via a monotonically increasing `last_seen_rank`) so re-confirmed symbols (e.g. current holdings) are never trimmed as stale. Discovered symbols are still subject to the signal computation's price/volume liquidity floor (see `decision-pipeline.md`) before they can ever be bought. A symbol that passes Alpaca's asset filter but comes back with zero historical bars (e.g. a thinly traded OTC ADR) can never clear that floor either, so it's excluded via `exclude_unpriceable()` for any such symbol that came from discovery (never a config-listed watchlist symbol, which stays eligible through a transient data outage) — persisted in `unpriceable_symbols` and filtered out of every future `next_batch()` result so the same dead symbol doesn't resurface and re-fail once the rotation cursor wraps back to it. Takes `asset_class` (default `"us_equity"`) and `symbol_filter` (default: tradable + fractionable + plain-ticker regex) constructor parameters; `CryptoRotationStrategy` passes `asset_class="crypto"` and `_crypto_asset_symbol_filter` (defined in `crypto_strategy.py`), since Alpaca's crypto assets come back as `"BASE/QUOTE"` pairs (e.g. `"BTC/USD"`) with no `fractionable` field, not plain tickers. Both strategies keep one instance cached per constructor-args key (`_autonomous_universe()`/`_crypto_autonomous_universe()`) instead of constructing a fresh one at each of the several call sites per iteration; schema creation only runs once per instance (`_schema_initialized`), same pattern as `TradeMemory` above.
- `ridge_regression.py` — the two-feature ridge fit both memory classes use.
- `signal_snapshot.py` — `build_snapshot_entries`/`write_snapshot`, a per-symbol opinion snapshot both strategies write near the end of their signal-evaluation loop (after posture-adjusted edge is known, before eligibility filtering trims to actionable candidates — see `decision-pipeline.md`). Purely observational, like the nightly pre-eval side channel. A write failure or an empty configured path is swallowed inside `write_snapshot` itself, matching `article_filter.py`'s "callers never need their own try/except" convention — this can never affect a trading decision. `scripts/view_signals.py`, a stdlib-only CLI table viewer, was removed once `scripts/web_dashboard.py` (below) superseded it.
- `trade_counter.py` — `record_trade(path, today)`, a one-function companion side channel to `signal_snapshot.py`: both strategy classes call it from their single order-submission choke point (`_submit_order_checked`) whenever the broker actually accepts an order, incrementing a same-day counter (`.portfolio_trade_count.json` / `.crypto_trade_count.json`) that resets instead of accumulating once the stored date is stale. Same fail-open posture as `signal_snapshot.py` — a read/write failure is swallowed, never raised — and it only ever counts trades after the fact; it cannot affect whether one is submitted.
- `scripts/web_dashboard.py` — stdlib-only (`http.server`, no Flask/no new pip dependency), installed by `setup_service.sh` as its own always-on unit, `trading-agent-dashboard.service`, separate from the two trading services. Reads only the four JSON files above (both snapshots, both trade counts) in response to an HTTP request — never touches the broker, never imports lumibot — and serves a single auto-polling HTML page (5s interval) showing each evaluated symbol's held/dip/edge/opinion plus a running "trades today" count. Binds `0.0.0.0:8765` by default (no authentication; LAN-reachable), configurable via `DASHBOARD_HOST`/`DASHBOARD_PORT` env vars set in its systemd unit. This is the superset replacement for the removed `scripts/view_signals.py` CLI script.

**Equity-only supporting modules** (no crypto equivalent yet)
- `news_context.py` — `WorldEventAnalyzer` fetches Alpaca news and produces a deterministic keyword score (`SEVERE_RISK_TERMS` −3, `RISK_TERMS` −1, `OPPORTUNITY_TERMS` +1, each counted once per article) wrapped in a `NewsContext`, which also carries the raw articles and a per-symbol score breakdown (from Alpaca's own article `symbols` tags) for downstream analysis. The pure `score_articles` classmethod (article dicts in, no network) does the scoring and is what `analyze()` calls after fetching; `NEWS_SCORE_REFINEMENT_ENABLED` (off by default) turns on recency decay and duplicate-phrase dampening inside it — off reproduces the original unweighted score exactly, so this can't silently change `NEWS_HIGH_RISK_SCORE`'s behavior. `NEWS_RSS_ENABLED` (off by default) merges `rss_news.py`'s free RSS headlines into the same `fetched_articles` list before `score_articles` runs — RSS articles carry no `symbols` tag, so they flow through the exact same untagged-mention text-scan `_symbol_news_scores` (`strategy.py`) already does for Alpaca. Merging happens only after Alpaca's own fetch succeeds without raising: a real Alpaca outage/credential problem still propagates and surfaces as `NewsContext.available = False` exactly as before RSS existed, so RSS can only add coverage, never mask a primary-source failure as false calm.
- `rss_news.py` — `fetch_articles(feed_urls, lookback_hours, max_articles)`, stdlib-only (no new dependency: `xml.etree.ElementTree` for RSS 2.0 parsing, `email.utils.parsedate_to_datetime` for `<pubDate>`, `requests` already a dependency). Each feed is fetched and parsed independently inside its own `try/except` — a down or slow publisher is logged and skipped, never taking the others down with it — then results are deduplicated by URL, filtered to `lookback_hours`, sorted newest-first, and capped at `max_articles` total. Verified against live feeds during development (Yahoo Finance, MarketWatch via `feeds.content.dowjones.io`, CNBC) — all three are plain public RSS with no API key, but free endpoints can change or disappear, so `NEWS_RSS_FEED_URLS` may need occasional maintenance, unlike Alpaca's contractually-stable API.
- `symbol_reference.py` — `SymbolReference`, a local ticker→company-name mapping cross-checked from Alpaca's per-symbol asset lookup and the SEC's public `company_tickers.json`, persisted in `.symbol_reference.duckdb` and refreshed at most every `SYMBOL_REFERENCE_REFRESH_DAYS`. `verified_symbols()` filters a spurious/malformed Alpaca symbol tag before it's trusted (empty result — nothing cached yet — means fail open, not reject-everything); `scan_text_for_symbols()` is a bounded company-name text scan, scoped only to the day's candidates, that catches a mention Alpaca's own tagging missed. Never creates a trade or veto on its own. `AssetRotationStrategy` keeps one instance cached per constructor-args key (`_symbol_reference()`) rather than constructing fresh each call, same instance-caching/schema-once pattern as `AutonomousUniverse` above.
- `llm_news.py` — optional `LLMNewsAnalyzer` sends the same articles to the loopback-only Ollama endpoint and returns a structured market-risk score from −10 to +10. Prompt and response budgets remain bounded for this Pi's CPU-only inference, and malformed replies are repaired defensively. When `LLM_NEWS_ENABLED` is true, `assess()` runs synchronously before order decisions; its threshold and fail-closed availability guard can veto new exposure. `check_red_flag()` likewise runs before orders for every negatively-covered discovery candidate and excludes severe company-specific risks. Only `summarize_day()` and `explain_exit()` are descriptive; exit notes remain deferred because the sale already occurred. The 09:00 warm-up and 8-hour keep-alive cover both daily trading windows.
- `article_filter.py` — standalone `extract_financial_context(url, watchlist)` fetches a candidate's linked article in full, retains its highest-signal sentences, and asks the same local model for structured sentiment, risks, and catalyst type. Results are cached per calendar day, URL, and watchlist digest in `.article_verdicts.duckdb`; the 03:00 nightly pass pre-warms that cache for managed/held symbols. During the live pre-trade discovery screen, a bearish verdict excludes the candidate before orders. Fetch/model failures remain contained inside this module and return no verdict.
- `adaptive_news_model.py` — `AdaptiveNewsModel`, a ridge-stabilized one-variable regression of next-day Asset B return on the news score. It only becomes authoritative after `NEWS_LEARNING_MIN_OBSERVATIONS` samples *and* sufficient `|correlation|`. Learns from Asset B as a market proxy inside portfolio mode. `strategy.py` instantiates it twice with separate state files: `_update_adaptive_learning` trains it on the deterministic keyword `news_context.score` (`.news_learning_state.json`, this is the one `_market_veto_reason` actually consumes via `NEWS_PREDICTED_RETURN_BLOCK_PERCENT`); `_update_llm_adaptive_learning` trains a second instance on `llm_assessment.score` (`.news_learning_state_llm.json`), purely so the two signals' predictive value can be compared over settled outcomes — its `llm_learned_forecast` is reported but not (yet) wired into any veto. Each instance keys one observation per calendar date regardless of how many times `on_trading_iteration` runs that day (`AdaptiveNewsModel.update`'s own same-day `pending` guard), so the twice-daily cadence does not double-weight a day's observation.

## Config

`main.py`'s `load_config` validates the entire `config.json` in one place — including all `CRYPTO_*` keys — even though only `main_crypto.py` consumes most of them. This is deliberate: a single validation implementation means the two processes can never silently disagree about what a config value means, and it's also why `build_strategy` (equity) threads `CRYPTO_ENABLED` into `AssetRotationStrategy`'s own parameters even though equity never trades crypto — see below. Cash allocation between the two pipelines is no longer a config value at all — each strategy class computes its own share at runtime: `CryptoRotationStrategy._account_half_value_dollars()` (`crypto_strategy.py`) always targets 50% of the shared account's live total value, since portfolio (equity) mode is never optional; `AssetRotationStrategy._crypto_reserve_dollars()` (`strategy.py`) targets the same 50% only when `crypto_enabled` is true, else zero — a crypto-disabled deployment (the shipped default) has nothing to reserve for a pipeline that never trades, and gets the full account instead of an unexplained half. Both sides source the raw total from their own `_account_total_value_dollars()`/inline equivalent, which treats a `None`, non-finite (NaN/inf), or negative broker read as zero (fail-open, never propagates a bad reading into the Decimal-based order-sizing math) and caches the result for `_ACCOUNT_VALUE_CACHE_SECONDS` (30s) since Lumibot's `get_portfolio_value()`/`get_cash()` each force their own fresh broker round-trip with no caching between them — without it, every buy attempt in a multi-candidate pass would pay for its own redundant network call (see `decision-pipeline.md`).
